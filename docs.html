Overview

BareBonesForms is a zero-build PHP form engine for shared hosting. Define a form as JSON, upload to hosting, embed it in your page, collect submissions. No npm. No framework. No build step. No mercy for unnecessary complexity.

What this engine provides:

  • Server-side validation — the client is for appearances; the server is the law
  • Email confirmation + admin notifications — so you know when someone knocks
  • Four storage backends — file, SQLite, MySQL, CSV — one config line, per-form override
  • Signed webhooks — HMAC-SHA256, because trust is a luxury
  • Custom post-submit actions — your PHP, your rules
  • Multi-page forms — split with page breaks, per-page validation
  • Conditional logic — show/hide fields based on other field values
  • Star ratings, sections, “other” option — the field types that matter
  • Sandbox mode — test everything without consequences
Not one for reading? Watch the wheels turn. If documentation is not how you prefer to spend your evening, go see it in action. Three live demos, each showing a different configuration — from the simplest arrangement to the full pipeline:

Come back here when you want to know why the wheels turn. Or don't. I ain't your schoolmaster.
Built for Plain PHP hosting, small and medium websites, and developers who want control without some bloated form system telling them how to live their life. Not built for drag-and-drop form building, enterprise workflow suites, or hoopleheads who need a GUI to tie their boots.

Quick Start

Five steps. A man of moderate intelligence could manage it before his coffee goes cold.

1. Upload the barebonesforms/ folder to your hosting.

2. Copy config.example.php → config.php and edit it. Once.

3. Open check.php in your browser — fix any errors it reports. Then delete check.php (it exposes server details to anyone who knows the URL).

4. Create a form JSON in forms/.

5. Embed it in your page:

html
<div data-form="kontakt"></div>
<script src="/barebonesforms/bbf.js"></script>

That's the full snippet. bbf.js auto-loads bbf.css from the same directory — no need for a separate <link> tag.

6. Done. The form renders, validates, and submits.

Form JSON Reference

Every form is a single JSON file in forms/. The forms/form.schema.json file provides IDE autocomplete and tooling validation — point your editor at it via "$schema": "form.schema.json". Runtime validation is a separate matter, handled by submit.php, which trusts no one and nothing.

json
{
    "$schema": "form.schema.json",
    "schema_version": 1,
    "id": "kontakt",
    "name": "Contact Form",
    "description": "Optional description shown above the form.",
    "submit_label": "Send",
    "submitting_label": "Sending…",
    "success_message": "Thank you!",

    "fields": [
        {
            "name": "email",
            "type": "email",
            "label": "Email",
            "placeholder": "john@example.com",
            "required": true,
            "description": "We'll never share your email."
        }
    ],

    "on_submit": {
        "store": true,
        "confirm_email": {
            "to": "{{email}}",
            "subject": "Thanks, {{name}}!",
            "template": "confirm.html"
        },
        "notify": {
            "to": ["admin@example.com", "backup@example.com"],
            "subject": "New contact: {{name}}",
            "template": "notify.html"
        },
        "webhooks": [
            "https://n8n.example.com/webhook/kontakt"
        ],
        "redirect": "https://example.com/thank-you",
        "actions": [
            { "type": "my_custom_action", "param": "value" }
        ]
    }
}

Advanced Features

Label Position

By default, labels sit above their fields. Set label_position at the form level to apply to all fields, or on individual fields for selective layout:

json
// Form-level — all fields
{ "label_position": "left", "fields": [ ... ] }

// Field-level — single field
{ "name": "qty", "type": "number", "label": "Quantity", "label_position": "left" }
ValueEffect
"top"Default — labels above inputs
"left"Labels to the left, inputs to the right

Input Prefix & Suffix

Attach text or unit labels to inputs:

json
{ "name": "price", "type": "number", "label": "Price", "suffix": "€" }
{ "name": "weight", "type": "number", "label": "Weight", "prefix": "~", "suffix": "kg" }

Renders as an input group with .bbf-input-prefix / .bbf-input-suffix spans. Can be combined with label_position and size.

Default Checked State

Pre-select radio or checkbox options using value on the field or checked on individual options:

json
// Radio — select by field value
{ "name": "shipping", "type": "radio", "value": "post",
  "options": [{"value": "post", "label": "Post"}, {"value": "pickup", "label": "Pickup"}] }

// Checkbox — option-level checked
{ "name": "extras", "type": "checkbox",
  "options": [{"value": "newsletter", "label": "Newsletter", "checked": true}] }

// Checkbox — select multiple by field value array
{ "name": "colors", "type": "checkbox", "value": ["red", "blue"],
  "options": ["red", "green", "blue"] }

Field Sizes

Control how wide a field renders. Useful when you don't need a full-width input for a ZIP code or age:

json
{ "name": "zip", "type": "text", "label": "ZIP Code", "size": "small" }
SizeWidth
"small"25% (min 120px)
"medium"50% (min 180px)
"large"Full width (default)

Column Layout

Radio buttons and checkboxes can be arranged in columns instead of a vertical stack:

json
{
    "name": "color",
    "type": "checkbox",
    "label": "Favorite colors",
    "columns": 3,
    "options": [
        {"value": "red", "label": "Red"},
        {"value": "blue", "label": "Blue"},
        {"value": "green", "label": "Green"},
        {"value": "yellow", "label": "Yellow"},
        {"value": "purple", "label": "Purple"},
        {"value": "orange", "label": "Orange"}
    ]
}

Values: 2 (two columns), 3 (three columns), "inline" (side by side, wrapping).

“Other” Option

Add a free-text “Other” option to radio, checkbox, or select fields. When selected, a text input appears for the user to type their answer:

json
{
    "name": "source",
    "type": "radio",
    "label": "How did you hear about us?",
    "other": true,
    "other_label": "Something else",
    "options": [
        {"value": "google", "label": "Google"},
        {"value": "friend", "label": "A friend"},
        {"value": "social", "label": "Social media"}
    ]
}

The submitted value contains the actual text the user typed, not the placeholder __other__ value. Server-side and client-side both resolve it.

Email Confirmation

Add "confirm": true to an email field to render a second “Confirm email” input. Both must match before submission:

json
{ "name": "email", "type": "email", "label": "Email", "confirm": true, "required": true }

Star Rating

The rating type renders an accessible star rating widget (1–5 by default). Fully keyboard-navigable with ARIA labels:

json
{ "name": "satisfaction", "type": "rating", "label": "How satisfied are you?", "required": true }

Use min and max to constrain the range if needed. The value is submitted as a number.

Section Breaks

Visual dividers to group related fields. Section fields are never submitted — they exist purely for layout:

json
{ "name": "section1", "type": "section", "title": "Personal Information" }

Multi-Page Forms

Insert page_break fields to split your form into pages. Previous/Next buttons appear automatically. Each page validates independently before advancing:

json
{
    "fields": [
        { "name": "name", "type": "text", "label": "Name", "required": true },
        { "name": "email", "type": "email", "label": "Email", "required": true },
        { "name": "page2", "type": "page_break" },
        { "name": "message", "type": "textarea", "label": "Message" }
    ]
}

The submit button only appears on the last page. Navigation button labels (“Next”, “Previous”) are translatable via language packs.

Conditional Logic

Show or hide fields based on another field's value. A hidden field is excluded from both validation and submission — as if it doesn't exist:

json
{
    "fields": [
        {
            "name": "contact_method",
            "type": "radio",
            "label": "Preferred contact method",
            "options": [
                {"value": "email", "label": "Email"},
                {"value": "phone", "label": "Phone"}
            ]
        },
        {
            "name": "phone_number",
            "type": "tel",
            "label": "Phone number",
            "show_if": { "field": "contact_method", "value": "phone" }
        }
    ]
}

The show_if object supports:

PropertyDescription
fieldName of the field to watch
valueValue or array of values that trigger visibility
opSet to "not" to invert — show when value does NOT match
Array values Pass an array to match multiple values: "value": ["email", "phone"] shows the field when either is selected. With "op": "not", it hides when any match.
Checkbox groups Conditions work with checkbox groups out of the box. If the source field is a checkbox group, the condition matches when any checked value equals the target. For example, "show_if": { "field": "skills", "value": "other" } triggers when "other" is among the checked options — regardless of what else is selected.

Animated Conditions

By default, conditional fields appear and disappear instantly. Add "animate_conditions": true at the form level to enable smooth expand/collapse transitions:

json
{
    "id": "my-form",
    "animate_conditions": true,
    "fields": [ ... ]
}

Fields slide in with a 300ms transition (max-height + opacity). The initial page load is always instant — no flickering. Only user-triggered changes animate.

Field Groups

The group type is a container that holds child fields. It renders a wrapper with an optional title and description. The key benefit: apply show_if to the group, and all children inherit the condition — show or hide an entire section with a single rule:

json
{
    "name": "assessment_group",
    "type": "group",
    "title": "Self-Assessment",
    "description": "Rate yourself honestly.",
    "show_if": { "field": "experience", "value": "0", "op": "gt" },
    "fields": [
        { "name": "rating_code", "type": "rating", "label": "Coding Skills", "required": true },
        { "name": "rating_teamwork", "type": "rating", "label": "Teamwork", "required": true },
        { "name": "cover_letter", "type": "textarea", "label": "Cover Letter", "required": true, "minlength": 50 }
    ]
}

When the group is hidden, its children are excluded from both validation and submission. Required fields inside a hidden group won't block the form. On the server side, the parent's show_if propagates to all children automatically.

Groups are containers, not fields A group itself is never submitted as data. Its children appear as top-level fields in the submission, just as if they weren't nested. Groups exist purely for conditional logic and visual structure.

Complex Conditions

Simple show_if checks one field against one value. For real-world forms, you often need compound logic — "show this if the user selected X and entered a number greater than Y." That's what all and any are for:

json
{
    "name": "senior_section",
    "type": "section",
    "title": "Senior Developer Details",
    "show_if": {
        "all": [
            { "field": "position", "value": ["frontend", "backend", "fullstack"] },
            { "field": "experience", "value": "5", "op": "gte" }
        ]
    }
}
CombinatorLogicDescription
allANDAll conditions must be true
anyORAt least one condition must be true

Combinators nest freely. You can put any inside all inside any — as deep as your logic demands:

json
{
    "show_if": {
        "all": [
            { "field": "gdpr", "value": "yes" },
            {
                "any": [
                    { "field": "experience", "value": "3", "op": "gte" },
                    { "field": "skills", "value": "react" }
                ]
            }
        ]
    }
}

This reads: "Show if GDPR is agreed and (experience ≥ 3 or React is selected)."

Comparison Operators

The op property in conditions supports more than just equals and not-equals. Use these operators for numeric comparisons, text searches, and presence checks:

OperatorMeaningExample use
(default)Equals{ "field": "color", "value": "red" }
notNot equals{ "field": "type", "value": "internal", "op": "not" }
gtGreater than{ "field": "age", "value": "18", "op": "gt" }
gteGreater or equal{ "field": "experience", "value": "5", "op": "gte" }
ltLess than{ "field": "score", "value": "50", "op": "lt" }
lteLess or equal{ "field": "budget", "value": "1000", "op": "lte" }
containsText contains{ "field": "notes", "value": "urgent", "op": "contains" }
emptyField is empty{ "field": "phone", "op": "empty" }
not_emptyField has value{ "field": "referral", "op": "not_empty" }
Numeric comparisons gt, gte, lt, lte parse both values as numbers. If either side isn't numeric, the condition evaluates to false. The value is always a string in JSON — the engine handles the conversion.
empty / not_empty These operators don't need a value property. For checkbox groups, empty means nothing is checked; not_empty means at least one option is selected.

Conditional Options

Individual radio/checkbox options can have their own show_if, letting you filter options dynamically without duplicating fields:

json
{
    "name": "shipping",
    "type": "radio",
    "label": "Shipping method",
    "options": [
        {"value": "post_sk", "label": "Slovenská pošta", "show_if": {"field": "country", "value": "SK"}},
        {"value": "post_cz", "label": "Česká pošta", "show_if": {"field": "country", "value": "CZ"}},
        {"value": "pickup", "label": "Osobný odber"}
    ]
}

Options without show_if are always visible. When an option is hidden, it is automatically unchecked.

Cross-Field Validation

Form-level validations array for rules that span multiple fields. Two rule types:

json
{
    "validations": [
        {
            "type": "min_sum",
            "fields": ["qty_book1", "qty_book2", "qty_book3"],
            "min": 1,
            "message": "Objednajte aspoň jednu knihu."
        },
        {
            "type": "min_filled",
            "fields": ["phone", "email"],
            "min": 1,
            "message": "Vyplňte telefón alebo email."
        }
    ]
}
TypeRule
min_sumSum of numeric field values must be ≥ min
min_filledAt least min fields must be non-empty

Validated both client-side (error in form message area) and server-side (422 response).

Date Validation

Date fields support min and max as date strings (ISO format). The browser's date picker respects them, and the server validates them:

json
{ "name": "event_date", "type": "date", "label": "Event Date", "min": "2026-01-01", "max": "2026-12-31" }

Readonly Fields

Set "readonly": true for fields that should be visible but not editable. Useful for pre-filled data:

json
{ "name": "ref", "type": "text", "label": "Reference", "value": "REF-2026-001", "readonly": true }

Custom CSS Classes

Add your own class to any field wrapper with css_class. Useful for custom styling without touching the global stylesheet:

json
{ "name": "vip_code", "type": "text", "label": "VIP Code", "css_class": "highlight-field" }

Reusable Templates

Define a set of fields once in the top-level templates object, then reference it from any group field with "use" and "prefix". The engine clones the template fields and prefixes all names automatically. Internal show_if references between template fields are prefixed too; references to form-level fields stay unchanged.

json
{
    "templates": {
        "modalities": [
            { "name": "time_worse", "type": "radio", "label": "When worse?", ... },
            { "name": "helps", "type": "checkbox", "label": "What helps?", ... }
        ]
    },
    "fields": [
        { "name": "areas", "type": "checkbox", "label": "Affected areas", ... },
        {
            "name": "head_group", "type": "group", "title": "Head",
            "show_if": { "field": "areas", "value": "head" },
            "use": "modalities", "prefix": "head_"
        },
        {
            "name": "stomach_group", "type": "group", "title": "Stomach",
            "show_if": { "field": "areas", "value": "stomach" },
            "use": "modalities", "prefix": "stomach_"
        }
    ]
}

This resolves to head_time_worse, head_helps, stomach_time_worse, stomach_helps — all from one template definition. See Demo 8 for a full example with 6 areas and 5 modality questions each.

PropertyOnDescription
templatesform rootObject of named template arrays. Each value is an array of field definitions.
usegroup fieldName of the template to clone as this group’s children.
prefixgroup fieldString prepended to every template field name (e.g. "head_").

on_submit Actions

The on_submit object governs what happens after a valid submission passes through the gates. Think of it as the chain of command — each action fires in order, and each does its appointed duty:

PropertyTypeDescription
storebooleanStore submission (default: true)
confirm_emailobjectSend email to respondent (to, subject, template)
notifyobjectSend email to admin (to — string or array of strings, subject, template)
webhooksarrayURLs to POST submission data to
redirectstringURL to redirect after submission
actionsarrayCustom server-side actions

All string properties support {{field_name}} interpolation. The subject line "New contact: {{name}}" becomes "New contact: John" at the moment of sending.

Embedding

Auto-init (simplest)

One script tag, one div. bbf.js auto-loads bbf.css from the same directory — no manual <link> needed.

html
<div data-form="kontakt"></div>
<script src="/barebonesforms/bbf.js"></script>

For those who prefer to hold the reins themselves:

html
<div id="my-form"></div>
<script src="/barebonesforms/bbf.js"></script>
<script>
    BBF.render('kontakt', '#my-form', {
        showTitle: false,
        hideOnSuccess: true
    });
</script>

Cross-Domain Embedding

When the form lives on one server and the page on another — a situation that requires careful handling, like doing business with a man you don't entirely trust.

baseUrl is required When embedding from a remote host, you must pass baseUrl pointing to your BareBonesForms installation. Without it, bbf.js will attempt to load form definitions from the current domain — which is not where they live. This will fail, and the form will not render.
html
<div id="my-form"></div>
<script src="https://forms.example.com/barebonesforms/bbf.js"></script>
<script>
    BBF.render('kontakt', '#my-form', {
        baseUrl: 'https://forms.example.com/barebonesforms/'
    });
</script>

Add the embedding origin to allowed_origins in config.php:

php
'allowed_origins' => ['https://www.mysite.com'],
CSRF + CORS CSRF tokens are automatically skipped for cross-origin requests (they're protected by CORS headers instead). Same-origin requests use session-based HMAC tokens. The system handles this distinction without your intervention.

Styling

bbf.css is auto-loaded by bbf.js — if the stylesheet sits next to the script, it will be injected into <head> automatically. No need to add a <link> tag manually (though you still can). To use your own styles instead, either override the classes below or delete bbf.css entirely.

Client-side

Load a language pack after bbf.js and set data-lang on your form container:

html
<script src="bbf.js"></script>
<script src="lang/de.js"></script>
<div data-form="kontakt" data-lang="de"></div>

Or pass the language in JavaScript:

js
BBF.render('kontakt', '#my-form', { lang: 'de' });

Multiple forms on one page can each speak a different language. A polyglot arrangement that would impress even the most seasoned saloon-keeper.

Server-side

Set in config.php:

php
'lang' => 'de',

Server-side validation errors will be returned in the configured language. Any key not found in the language file falls back to English. No missing messages, no broken forms — the system is more forgiving than I am.

Adding a Custom Language

Create two files in the lang/ directory. Use lang/en.js and lang/en.php as your reference — they contain every key.

Client-side (lang/xx.js) — 25 translatable keys:

js
BBF.registerLang('xx', {
    loading:       'Loading…',
    submitDefault: 'Submit',
    required:      '{label} is required.',
    invalidEmail:  '{label} must be a valid email.',
    // … see lang/en.js for all keys
});

Server-side (lang/xx.php) — 14 validation message keys:

php
<?php
return [
    'required'     => '{label} is required.',
    'invalidEmail' => '{label} must be a valid email.',
    // … see lang/en.php for all keys
];

Placeholders use {label}, {min}, {max}, {status}, {id}, {message} syntax. The system replaces them at runtime. If you forget a key, the English default takes over. If you misspell a placeholder, well — that's on you.

Storage Backends

Four backends. One config line. Choose your weapon:

php
'storage' => 'file',   // or 'sqlite', 'mysql', 'csv'

File storage (default)

JSON files in submissions/{form_id}/. Zero configuration. Works everywhere. File locking for race conditions. The simplest arrangement — and often the wisest.

SQLite

Single file database. Zero config, SQL capable. Auto-creates on first submission. WAL mode for concurrency. For a man who wants the power of SQL without the burden of running a database server.

php
'storage' => 'sqlite',
'sqlite'  => ['path' => __DIR__ . '/submissions/bbf.sqlite'],

MySQL / MariaDB

Set 'storage' => 'mysql' and fill in credentials. Table bbf_submissions auto-created. For high-volume operations where file-based storage starts sweating.

CSV

One CSV per form in submissions/{form_id}.csv. Opens in Excel, Google Sheets, or whatever spreadsheet your accountant swears by. File-locked writes. Column order from form definition.

A word of candor about CSV Multi-checkbox values are joined with ; in CSV cells. CSV is best understood as human-readable export storage, not a fully reversible archive. If that bothers you, use SQLite. I won't judge.

Comparison

File (JSON)SQLiteMySQLCSV
SetupZero configZero configRequires DBZero config
Scale< 1k/form< 100k/formAny volume< 5k/form
QueriesBasicFull SQLFull SQLSequential
Best forPrototypesMedium sitesProductionSpreadsheets

Per-form storage override

The global storage setting in config.php applies to all forms by default. But you can override it for individual forms by adding "storage" to the form JSON:

json
{
    "id": "survey",
    "storage": "sqlite",
    "fields": [ ... ]
}

This form will store to SQLite while all others continue using the global backend. Accepted values: file, csv, sqlite, mysql. The corresponding backend configuration (SQLite path, MySQL credentials) is still read from config.php.

Submissions API

Every submission that passes through this system is yours to query. But first — authentication. I don't let strangers walk behind my bar, and neither should you.

Set api_token in config.php:

php
'api_token' => 'your-secret-token',

Pass the token via header (recommended — keeps it out of server logs and browser history, where any passing scoundrel might see it):

bash
# Recommended: header auth (keeps the token where it belongs — out of sight)
curl -H "X-BBF-Token: your-secret-token" "submissions.php?form=kontakt"

# Fallback: query param (acceptable, if you must)
GET submissions.php?form=kontakt&token=YOUR_TOKEN

# Pagination
GET submissions.php?form=kontakt&token=YOUR_TOKEN&limit=20&offset=40

# Date filter
GET submissions.php?form=kontakt&token=YOUR_TOKEN&from=2024-01-01&to=2024-12-31

# Single submission
GET submissions.php?form=kontakt&id=bbf_a1b2c3d4&token=YOUR_TOKEN

# Export CSV
GET submissions.php?form=kontakt&format=csv&token=YOUR_TOKEN

# Quick export with ?last= shorthand
GET submissions.php?form=kontakt&format=csv&token=YOUR_TOKEN&last=7d
GET submissions.php?form=kontakt&format=csv&token=YOUR_TOKEN&last=24h
GET submissions.php?form=kontakt&format=csv&token=YOUR_TOKEN&last=50

Quick export: ?last=

The last parameter is a shorthand for common date/count filters. Saves you from calculating dates at 2 AM:

ValueMeaningExample
7dLast 7 days?last=7d
24hLast 24 hours?last=24h
2wLast 2 weeks?last=2w
3mLast 3 months?last=3m
50Last 50 submissions?last=50

Works with both JSON and CSV format. When last is a time-based value, it sets from automatically. When it's a plain number, it sets limit. If from is already provided, last is ignored.

Response format

json
{
    "form": "kontakt",
    "returned": 20,
    "total": 142,
    "limit": 20,
    "offset": 40,
    "submissions": [
        {
            "id": "bbf_a1b2c3d4e5f6",
            "form": "kontakt",
            "data": { "name": "John", "email": "john@example.com" },
            "meta": { "submitted": "2024-06-15T10:30:00+02:00", "ip": "..." }
        }
    ]
}
FieldDescription
returnedNumber of submissions in this page
totalTotal matching submissions across all pages
limitMax items per page (default: 100, max: 1000)
offsetNumber of items skipped

Email Templates

HTML templates in templates/ use {{field_name}} placeholders. All values are HTML-escaped, because a man who doesn't sanitize his inputs deserves every injection he gets.

VariableValue
{{_form}}Form name
{{_id}}Submission ID
{{_time}}Submission timestamp
{{_summary}}HTML table of all fields
{{field_name}}Any form field value

Conditional Sections

Show or hide parts of a template based on whether a field has a value:

html
{{#quantity}}Ordered: {{quantity}} ks{{/quantity}}

{{^gift_note}}No gift note provided.{{/gift_note}}
SyntaxMeaning
{{#var}}...{{/var}}Show block only if var is truthy (non-empty)
{{^var}}...{{/var}}Show block only if var is falsy (empty)

Unreplaced {{tags}} are automatically removed from the output.

Option Labels in Templates

For select/radio/checkbox fields, the raw value is available as {{fieldname}} and the display label as {{fieldname_label}}:

html
Payment: {{payment_method_label}}   →  "Cash on delivery (+€1)"
Shipping: {{shipping_label}}        →  "Slovenská pošta"
UTF-8 support Email subjects and sender names with diacritics are automatically RFC 2047 encoded (both SMTP and mail()). Because we live in a world where people have names with accents, and any system that cannot handle that is not fit for purpose.

Custom Actions

A man sometimes needs to do things his own way. Create a PHP file in actions/ and reference it in your form JSON:

json
"actions": [
    { "type": "log_to_crm", "crm_url": "https://..." }
]
php
// actions/log_to_crm.php
// Available variables: $action, $submission, $config

$url = $action['crm_url'] ?? '';
if ($url) {
    file_get_contents($url . '?' . http_build_query($submission['data']));
}
Trusted code only — and I shall say this once Actions run as PHP includes with full server access. They are not sandboxed. Only place code you trust in the actions/ directory. If you let a stranger put files there, whatever follows is on your own head.

Security Model

The world is full of those who would exploit what you build. BareBonesForms does not pretend this isn't so. Sensible defaults for shared PHP hosting — here is how we maintain order in this camp:

ProtectionHow
CSRF tokensSession-based HMAC for same-origin. Automatic via bbf.js.
HoneypotHidden field catches bots. They walk right into it, the poor stupid creatures.
Rate limitingFile-locked per-IP counter. Default: 10/minute.
Server-side validationAll rules enforced on server. The client's opinion is noted and ignored.
Input sanitizationhtmlspecialchars() in all email templates. No exceptions.
Email header injectionTo/Subject sanitized, emails validated.
Signed webhooksHMAC-SHA256 via X-BBF-Signature.
CORS controlConfigurable allowed_origins. Empty = same origin only.
API authenticationToken required. Disabled until you configure it.
SQL injectionPDO prepared statements. Every single one.
Directory access.htaccess blocks 7 directories, all config files, editor backups, and dotfiles.
CSV formula injectionCells starting with = + - @ are prefixed. Your spreadsheet is safe.
Config protectiondefined('BBF_LOADED') || exit; guard — even if .htaccess fails, direct access outputs nothing.
Template traversalbasename() on all template paths — ../../config.php in form JSON is neutralized.
SSRF protectionWebhook URLs validated against private/reserved IP ranges (127.x, 10.x, 172.16.x, 192.168.x, 169.254.x).
Error suppressiondisplay_errors forced OFF — PHP errors logged to server, never shown to browsers.
Definition strippingForm definition API strips on_submit and storage — webhook URLs, emails, and actions are not exposed.

What .htaccess blocks

The included .htaccess is comprehensive. Here is what it denies:

  • Directories: submissions/, logs/, templates/, actions/, backups/
  • Files: config* (all variants), forms/*.json, lang/*.php
  • Editor backups: *.bak, *.swp, *.save, *.orig, *~
  • Dotfiles: .git/, .env, .htpasswd — everything starting with a dot
  • Directory listing: OFF globally (Options -Indexes)

Web server compatibility

The .htaccess works on Apache only. If you run Nginx or LiteSpeed, you need equivalent rules — fail to add them and your credentials, submissions, and templates will be accessible to anyone who guesses the URL:

nginx
# Add to your server {} block
location ~ ^/config        { deny all; return 403; }
location ~ ^/(submissions|logs|templates|actions|backups|lang)/ { deny all; return 403; }
location ~ /forms/.*\.json$ { deny all; return 403; }
location ~ /\.(git|env|htaccess|htpasswd) { deny all; return 403; }
autoindex off;

Daily security self-check

submit.php runs a lightweight self-check once every 24 hours. Non-blocking, log-only — no forms affected, no users aware. It writes warnings to PHP's error_log when it detects:

  • check.php still exists on the server
  • Sandbox mode is ON
  • api_token is empty
  • webhook_secret is empty and at least one form uses webhooks
  • .htaccess is missing
  • display_errors is ON in PHP config

Check your PHP error log periodically. Or set up log monitoring. The warnings will be there — waiting patiently for someone responsible to read them.

check.php — installation diagnostics

Run check.php after installation. It verifies your setup and actively probes for security issues:

  • PHP version, extensions, and configuration
  • Storage backend connectivity
  • Form JSON validity and field definitions
  • Active HTTP probes on submissions/, logs/, templates/, actions/, forms/ — verifies each one is actually blocked
  • config.php accessibility via HTTP
  • BBF_LOADED guard presence in config.php
  • display_errors state
  • Sandbox mode state
  • Leftover diagnostic files (phpinfo.php, test.php, etc.)

Access: localhost = unrestricted. Remote = requires ?token=<api_token>.

Delete check.php after verification. It exposes PHP version, loaded extensions, directory paths, storage backend details, and form structure. Even with token protection, this file has no business existing on a production server.

Privacy

By default, BareBonesForms stores IP address and user-agent with each submission. If your conscience — or your lawyer — demands otherwise:

php
'store_ip'         => false,  // stop storing IP addresses
'store_user_agent' => false,  // stop storing user-agent strings

Submissions always store the timestamp. Form field data is stored as submitted. What you do with it is your affair.

Webhooks

When webhook_secret is set in config.php, every webhook POST carries a signature — so the receiving end knows the message is genuine and not some impostor's forgery:

http
X-BBF-Signature: sha256=<hex-encoded-hmac>

Verify on the receiving end. Trust, but verify. Actually — just verify:

php
$payload = file_get_contents('php://input');
$expected = 'sha256=' . hash_hmac('sha256', $payload, 'your-webhook-secret');
$received = $_SERVER['HTTP_X_BBF_SIGNATURE'] ?? '';
if (!hash_equals($expected, $received)) {
    http_response_code(403);
    exit('Invalid signature');
}

Sandbox Mode

A man ought to test his arrangements before the stakes are real. Enable in config.php:

php
'sandbox' => true,

Then open sandbox.php. A dry run. A rehearsal before the curtain rises. The sandbox:

  • Renders any form from forms/
  • Runs full server-side validation
  • Shows detailed validation results per field
  • Previews email content (confirmation + notification) with rendered templates
  • Lists webhook URLs that would be called
  • Shows custom action files and whether they exist
  • Does NOT store submissions, send emails, fire webhooks, or run actions
Disable before going public. Set 'sandbox' => false before production. The sandbox exposes form definitions, webhook URLs, email templates, and action file existence. On remote servers, sandbox access requires ?token=<api_token> — but even so, disable it when you're done testing.

Error Notifications

When form processing fails — storage, email, or webhook errors — the admin can be notified by email. Set error_notify in config.php:

php
'error_notify' => 'admin@example.com',

Notifications are throttled to at most one email per 24 hours to prevent inbox flooding. The notification includes the form ID, error type, and a timestamp. Leave empty (the default) to disable.

The notification is sent via PHP mail() directly (not SMTP), so it works even when SMTP itself is the cause of the failure. Errors are always logged to PHP error_log regardless of this setting.

Production Checklist

Before you open the doors to the public, attend to every item on this list. Skip one, and you'll regret it. I've seen it happen.

  • Copy config.example.php → config.php and fill in all settings
  • Set a strong api_token for submissions API access
  • Set webhook_secret if using webhooks
  • Configure SMTP for reliable email delivery (mail.method → smtp)
  • Verify .htaccess works (or add Nginx/LiteSpeed equivalent)
  • Test with HTTPS — all forms should run over TLS
  • Review form definitions with form.schema.json validation
  • Set appropriate rate_limit for your traffic
  • Verify submissions/ and logs/ are not web-accessible
  • Set allowed_origins if embedding cross-domain
  • Review privacy settings: store_ip / store_user_agent
  • Set error_notify to receive alerts when form processing fails
  • Disable sandbox mode: 'sandbox' => false
  • Run check.php to verify installation (remote access requires ?token=<api_token>)
  • Delete check.php after verification — it exposes PHP version, extensions, directory paths, and configuration details
  • Delete or protect editor.php — can modify form definitions on the server
  • Delete or protect viewer.php — exposes submission data to anyone with the token
